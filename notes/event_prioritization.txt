
Given that systems can generate thousands of logs per hour.

It is important to have a system that prioritizes those that require an immediate response
over those that are nonurgent or informational only.

On Windows, system and application events are classified as:

	- Informatinal
	- Warning
	- Critical

Audit logs only record Success/Fail events.

Syslog Severity Levels are at follows:

- [0]Emergency: system is unusable (kernel panic)

- [1]Alert: a fault requiring immediate remediation.
- [2]Critical: a fault that will require immediate remediation is likely to develop.

- [3]Error: a non-urgent fault has developed 
- [4]Warning: a non-urgent fault is likely to develop

- [5]Notice: a state that could lead to a possible error condition has developed	
- [6]Informational: a normal but reportable event has occurred.

- [7]Debug: Verbose status used during development and testing.

Logging level determines the maximum level at which events are recorded or formatted.
For example if logging level of remote forwarding is set to for. Levels 5, 6, 7 will not be forwarded.

An automated management system can be set to generate some sort of alert that can indicate:

	- Whenever certain type of events of a given severity are encountered.

An alert can also be generated by setting thresholds for perfomance encounters, such as:

	- Packet Loss	- Link Bandiwth Drops	- Number of Sessions Established
	- Delay/Jitter time in real-time applications	- And so on...

Difference between alert and notification:

Alert: Matched some sort of pattern or filter that should be recorded or highlighted.

Notification: System sends a message to notify the occurrence of the alert.
	- Low-priority alert:	may simply be displayed in the system dashboard
	- High-priority alert:	may use some sort of active-notification messaging, such as
				as SMS, email,  or physical alarm signal.
